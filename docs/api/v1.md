# OpenBox API v1

The canonical machine-readable contract is [`api/openapi.yaml`](../../api/openapi.yaml).
This page describes operational behavior that is easy to miss when reading the
schema alone.

## Compatibility

Clients send `X-OpenBox-API-Version: v1`. The daemon returns the same header on
v1 responses and returns an upgrade-required error for an unsupported version.
Clients should ignore additive JSON fields, but must report unknown enum values
with the resource and field that contained them.

## Durable mutations

Instance create, start, stop, restart, and delete require an
`Idempotency-Key`. They return a durable operation promptly; Incus work is done
later by the daemon worker. Retrying the identical request with the same key
returns that operation. Reusing the key for different work returns
`idempotency_conflict`.

An HTTP disconnect does not cancel an operation. Query
`GET /v1/operations/{id}` or follow
`GET /v1/operations/{id}/events` to observe completion.

## Event streams

Operation events are Server-Sent Events ordered by their integer sequence.
Reconnect with the last sequence in `Last-Event-ID`; the daemon replays newer
durable events before following live progress. Heartbeat comments keep idle
connections observable. A terminal operation event ends the stream.

Closing the stream only stops delivery. It never changes operation state.

## Cancellation

`POST /v1/operations/{id}/cancel` succeeds only before a worker claims the
operation or its stage advances beyond `runtime`. Once external work may have
started, the response is `cancellation_unsafe`; inspect the operation rather
than assuming it was rolled back.

## Errors

Errors use a stable envelope containing a machine-readable code, safe message,
optional field and retryability, plus a request ID. Internal SQL, Incus socket,
filesystem, and operation-payload details are not part of the public response.

## Owner authentication

Health, bootstrap status/consume, and session login are the only public v1
routes. Every other route requires either the host-only browser session cookie
or an `Authorization: Bearer` owner token. The authenticated principal supplies
the owner identity; request bodies cannot select another owner.

Cookie-authenticated unsafe methods require the per-session value in
`X-CSRF-Token`. Login, setup, and `GET /v1/session` return that value in the JSON
session response. Session refresh rotates both the cookie and CSRF secret while
preserving the absolute expiry. Bearer requests do not require CSRF.

Owner tokens are returned only by `POST /v1/tokens`. `GET /v1/tokens` exposes
metadata without the secret or digest, and `DELETE /v1/tokens/{token_id}` makes
the next use fail. SSH public keys are normalized and owner-scoped under
`/v1/ssh-keys`.

See [the owner authentication security model](../security/owner-authentication.md)
for bootstrap, password, session, transport, and key-handling decisions.
