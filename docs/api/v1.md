# OpenBox API v1

The canonical machine-readable contract is [`api/openapi.yaml`](../../api/openapi.yaml).
This page describes operational behavior that is easy to miss when reading the
schema alone.

## Compatibility

Clients send `X-OpenBox-API-Version: v1`. The daemon returns the same header on
v1 responses and returns an upgrade-required error for an unsupported version.
Clients should ignore additive JSON fields, but must report unknown enum values
with the resource and field that contained them.

## Durable mutations

Instance create, start, stop, restart, and delete require an
`Idempotency-Key`. They return a durable operation promptly; Incus work is done
later by the daemon worker. Retrying the identical request with the same key
returns that operation. Reusing the key for different work returns
`idempotency_conflict`.

An HTTP disconnect does not cancel an operation. Query
`GET /v1/operations/{id}` or follow
`GET /v1/operations/{id}/events` to observe completion.

## Event streams

Operation events are Server-Sent Events ordered by their integer sequence.
Reconnect with the last sequence in `Last-Event-ID`; the daemon replays newer
durable events before following live progress. Heartbeat comments keep idle
connections observable. A terminal operation event ends the stream.

Closing the stream only stops delivery. It never changes operation state.

## Cancellation

`POST /v1/operations/{id}/cancel` succeeds only before a worker claims the
operation or its stage advances beyond `runtime`. Once external work may have
started, the response is `cancellation_unsafe`; inspect the operation rather
than assuming it was rolled back.

## Errors

Errors use a stable envelope containing a machine-readable code, safe message,
optional field and retryability, plus a request ID. Internal SQL, Incus socket,
filesystem, and operation-payload details are not part of the public response.

## Owner authentication

Health, bootstrap status/consume, session login, the certificate allow ask, and
gateway auth are the only public v1 routes. Every other route requires either
the host-only browser session cookie or an `Authorization: Bearer` owner token.
The authenticated principal supplies the owner identity; request bodies cannot
select another owner.

Cookie-authenticated unsafe methods require the per-session value in
`X-CSRF-Token`. Login, setup, and `GET /v1/session` return that value in the JSON
session response. Session refresh rotates both the cookie and CSRF secret while
preserving the absolute expiry. Bearer requests do not require CSRF.

Owner tokens are returned only by `POST /v1/tokens`. `GET /v1/tokens` exposes
metadata without the secret or digest, and `DELETE /v1/tokens/{token_id}` makes
the next use fail. SSH public keys are normalized and owner-scoped under
`/v1/ssh-keys`.

## Sandboxes

Instances with `kind: sandbox` are disposable environments with a mandatory
TTL, default restricted egress, and argv exec. Create/start/stop/delete still
use durable operations; exec and TTL extension are synchronous.

See [Sandbox lifecycle](sandbox-lifecycle.md) for the state machine, isolation
selection (`strong` vs `container` vs omitted), operation stages, NDJSON exec
frames, concurrency/rate limits, expiry/extension rules, cleanup failure
fields, and the machine-readable error catalog.

## Checkpoint and reuse

Named disk-only checkpoints, restore-as-new, and live clone are durable
mutations with `Idempotency-Key` and operation progress under
`/v1/operations`. Derived instances get rewritten ownership metadata,
re-personalized OpenBox-managed SSH access, and provenance fields. Copy-on-write
is claimed only when runtime capabilities confirm a CoW-capable storage driver.

See [Checkpoint and reuse](checkpoint-and-reuse.md) for operation shapes,
warnings, CoW status vocabulary, credential policy, and CLI mapping. SSH
gateway `cp` is not a supported product surface.

## HTTPS routes

Routes are created explicitly and are always private by default:

- `GET /v1/routes` / `POST /v1/routes` — list and create routes
- `GET|PATCH|DELETE /v1/routes/{route_id}` — read, update hostname/port, or delete
- `POST /v1/routes/{route_id}/publish` — explicit visibility flip to public
- `POST /v1/routes/{route_id}/validate-dns` — check DNS and set `tls_state` to
  `pending`, `invalid`, or `active`

`GET /v1/instances/{instance_id}/suggested-ports` returns candidate ports for an
instance without creating routes. When the runtime cannot list listening ports
yet, `items` is an empty array. Suggestions never auto-publish.

`GET /v1/certificates/allow?domain=HOSTNAME` is the Caddy on-demand TLS ask
endpoint. It returns 200 only when the hostname matches a persisted route
(private or public). It is unauthenticated by design; bind the API privately so
only the local gateway can call it.

`GET /v1/gateway/auth` is the Caddy forward_auth check. Public routes return 200
without credentials. Private routes require an owner session cookie, owner API
bearer token, or a route-scoped bearer token (`obr_…`).

See [the owner authentication security model](../security/owner-authentication.md)
for bootstrap, password, session, transport, and key-handling decisions.
