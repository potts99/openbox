# OpenBox HTTPS gateway (Caddy)

OpenBox runs a separate, tightly scoped Caddy process for HTTPS routing.
`openboxd` generates the managed route snippet from **approved database routes
only** and applies it atomically (`internal/caddy`).

## Layout

| Path | Purpose |
|------|---------|
| `Caddyfile` | Operator-owned base config (TLS, logging, include) |
| `routes.caddyfile` | Generated by OpenBox; do not edit by hand |
| [`marketing/`](../../marketing/) | Standalone public landing site (not embedded in `openboxd`) |

Example base `Caddyfile`:

```caddyfile
{
	on_demand_tls {
		ask http://127.0.0.1:8443/v1/certificates/allow
	}
	# email admin@example.com
}

# Owner console + API (embedded dashboard)
app.example.com {
	reverse_proxy 127.0.0.1:8443
}

# Public marketing site — separate from the console binary
example.com, www.example.com {
	root * /var/www/openbox-marketing
	file_server
	encode gzip
}

import routes.caddyfile
```

Deploy marketing files with `rsync -a --delete marketing/ host:/var/www/openbox-marketing/`.
Do not serve the marketing site from the `openboxd` process.

The ask URL hits OpenBox's public (unauthenticated) allow endpoint. Bind the
API to loopback so only the local Caddy process can request certificates.

## Apply semantics

1. Write a temp file next to `routes.caddyfile`
2. Validate the candidate (`caddy validate --config …`)
3. Rename into place
4. Reload Caddy
5. On reload failure, restore the previous file and reload again

Private-route auth uses `forward_auth` to `http://127.0.0.1:8443/v1/gateway/auth`.

## Failure independence

If Caddy validate/reload fails, OpenBox rolls back the generated snippet and
leaves instances running. Instance lifecycle and host SSH (port 22) are not
driven by the HTTPS gateway process. Keep the OpenBox SSH gateway on port 2222
(default) so host recovery SSH remains available when either OpenBox gateway
fails.
